# AutoApply Bot

Multi-user Telegram bot for automated job discovery + recruiter outreach.

Sibling project to [`AutoApply`](https://github.com/codebyanjaneya/autoapply) (the single-user CLI). Different architecture (async, Postgres, multi-tenant), shared business logic to be extracted later.

## Week 1 status: schema + foundations

This commit contains:

- `core/models.py` — SQLAlchemy 2.0 ORM models. Every per-user table starts with `user_id` and has it indexed. Tables: `users`, `user_preferences`, `user_credentials`, `subscriptions`, `jobs`, `applications`, `outreach_logs`, `daily_run_summary`, `rate_limit_counters`, `stripe_events`.
- `core/db.py` — async engine + `get_session()` context manager.
- `core/tenant.py` — `TenantContext` dataclass + `load_tenant_context()` loader. Every worker function takes this explicitly; no globals.
- `core/crypto.py` — Fernet wrapper for SMTP password + Hunter key at rest.
- `docker-compose.yml` — local Postgres 16 (optional — see "Postgres options" below).
- `requirements.txt`, `.env.example`, `.gitignore`.

Not yet present (later weeks): Telegram handlers, Adzuna scraper, Groq scorer, SMTP sender, Stripe webhooks, APScheduler workers.

## Postgres for local dev — pick one

You do **not** need Docker. Any of these work:

| Option | Setup time | Notes |
|---|---|---|
| **Neon free tier** (recommended) | 2 min | https://neon.tech — 0.5 GB free, branching for safe migration testing, just paste the connection string into `DATABASE_URL` |
| **Supabase free tier** | 2 min | https://supabase.com — 500 MB, includes dashboard |
| **Railway Postgres** | 2 min | Same provider as your eventual production DB — keeps dev/prod parity |
| **Docker Desktop** | 10 min install | Use the bundled `docker-compose.yml`; `docker compose up -d` |
| **Native Postgres install** | 15 min | `winget install PostgreSQL.PostgreSQL` then create user/db manually |

**Recommendation: Neon.** It's free, instant, and the branching feature is unmatched for trying scary migrations — make a branch, run `alembic upgrade head`, if it breaks, delete the branch.

## Local bootstrap

```powershell
# 1. Python environment
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

# 2. Postgres — pick ONE of these:

# 2a. Neon / Supabase / Railway: just paste the connection string into .env
#     (no local install needed)

# 2b. Docker (if installed):
docker compose up -d

# 3. Environment
Copy-Item .env.example .env
# then fill in TELEGRAM_BOT_TOKEN, GROQ_API_KEY, etc.
# generate ENCRYPTION_KEY with:
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"

# 4. Apply the initial migration
#    alembic.ini + migrations/ are already in the repo — do NOT run `alembic init`.
alembic upgrade head
```

To verify:

```powershell
alembic current     # -> 0001 (head)
alembic history     # -> 0001 -> initial schema
```

To roll back during development:

```powershell
alembic downgrade base    # drops every table + enum type
alembic upgrade head      # re-applies
```

### Writing new migrations

Migrations are **hand-written**, not autogenerated. Reasons: explicit control over enum types, server defaults, and index naming; ability to add data-migration steps cleanly; reviewable diffs.

```powershell
alembic revision -m "add followups table"
# then edit migrations/versions/000N_add_followups_table.py manually
```

## Architecture rules

1. **Every database query filters on `user_id`.** If a function reads or writes per-user tables, it must take `TenantContext`. Reviewers should reject any PR that violates this.
2. **No plaintext secrets in the database.** SMTP passwords and Hunter keys go through `core.crypto.encrypt()` before storage.
3. **Job-board access uses first-party APIs only.** Primary: Adzuna (`/v1/api/jobs/in/`). Fallback: JSearch via RapidAPI. No scraping of LinkedIn / Indeed / Naukri.
4. **Free tier limits are enforced at the worker layer** via `rate_limit_counters`. Before every outreach send / scan call, the pipeline does an atomic `INSERT … ON CONFLICT DO UPDATE` to increment the counter and check it against `UserPreferences.daily_outreach_limit` / `daily_scan_limit`. Stripe webhook updates `User.subscription_tier` and bumps the preference limits accordingly.
5. **Stripe webhooks are idempotent via `stripe_events`.** Every handler INSERTs the `event_id` as its first step inside the transaction — a duplicate retry from Stripe fails the PK constraint, rolls back, and returns 200 OK. No double charges, no double upgrades.
